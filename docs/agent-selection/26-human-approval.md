## Prompt 不是审批

“请在高风险操作前询问用户”不是可靠安全机制。真正的人机确认必须由运行时强制执行。

```text
Risk Classifier
  -> Pause
  -> Show Plan / Diff / Impact
  -> User Approval
  -> Execute Tool
  -> Audit Log
```

模型可以提出操作计划，但不能绕过确认点。

## 确认点是状态，不是弹窗

高风险流程里，确认点应该进入状态机：

```text
planned
  -> pending_approval
  -> approved / rejected
  -> executed
  -> audited
```

如果确认结果没有持久化，任务中断后就无法判断是否已经批准。

## 哪些动作需要确认

默认需要确认：

- 删除；
- 支付；
- 发送外部消息；
- 发布内容；
- 修改生产数据；
- 执行 shell；
- 大范围导出；
- 权限变更；
- 自动合并代码。

如果操作不可逆、影响外部用户、涉及资金或权限，就必须确认。

## 风险分级

| 风险 | 例子 | 处理 |
| --- | --- | --- |
| 低 | 只读查询、草稿生成 | 可自动 |
| 中 | 修改草稿、创建待审批记录 | 可批量确认 |
| 高 | 发送、发布、支付、删除 | 单次确认 |
| 极高 | 权限变更、生产数据批量修改 | 双人审批或禁止自动化 |

风险分级应该由程序计算，不应该只由模型判断。

## 确认页应该展示什么

至少展示：

- 即将执行的动作；
- 目标对象；
- 参数；
- 影响范围；
- 风险等级；
- 可回滚性；
- 引用或依据；
- 执行后日志位置。

不要只问“是否继续”。用户需要知道继续的具体含义。

## 一个典型确认流

假设客服 Agent 准备给用户补发优惠券。模型可以生成计划，但真正执行前，运行时应该把计划固化成可审查对象：

```text
intent: 补发优惠券
target: user_id=8421
amount: 50 元
reason: 订单延迟超过 SLA
risk: medium
rollback: 可在 24 小时内撤销
evidence: 工单、订单状态、SLA 规则
```

用户确认的不是一句自然语言，而是这份结构化计划。执行时工具只能使用被批准的参数，不能让模型在审批后临时改金额、换用户或增加额外动作。

如果执行需要多步，比如“查询订单 -> 创建补偿记录 -> 发送通知”，审批应该绑定整份计划和每个关键参数。任何高风险参数变化，都要回到 `pending_approval`，不能沿用旧批准。

## 审批后还要限制执行

人工点了同意，不代表后续所有动作都放开。审批记录应该约束：

- 哪个用户批准；
- 批准了哪个计划版本；
- 批准时间；
- 批准的工具和参数；
- 有效期；
- 是否允许重试；
- 失败后是否需要重新确认。

最常见的漏洞是“先审批一个安全计划，执行时模型又生成了新参数”。解决办法是把 approval id 传给工具网关，由工具网关校验本次工具调用是否和已批准计划一致。

## 审批粒度

| 粒度 | 适合 |
| --- | --- |
| 单次操作确认 | 高风险动作 |
| 一批操作确认 | 多个同类低风险动作 |
| 会话授权 | 低风险只读操作 |
| 策略授权 | 固定自动化任务 |

授权越宽，审计和撤销机制越重要。

## 常见错误

| 错误 | 后果 |
| --- | --- |
| 只在 Prompt 里要求确认 | 模型可能漏问或问错 |
| 确认页只显示一句话 | 用户不知道影响范围 |
| 审批结果不持久化 | 中断恢复后无法判断状态 |
| 审批和工具参数脱节 | 批准 A，执行 B |
| 批量确认没有上限 | 一次误批影响范围过大 |
| 没有撤销和审计 | 出错后无法追责和修复 |

审批设计的重点不是让用户多点一次按钮，而是把高风险动作从“模型自由发挥”变成“程序可验证的状态迁移”。

## LangGraph 的价值

如果流程需要暂停、恢复、审批后继续执行，显式状态图会更合适：

- 保存计划；
- 保存审批状态；
- 从确认节点继续；
- 失败后回到指定节点；
- 回放每一步。

轻量工具调用也能做确认，但复杂流程更适合状态机。

## 上线检查

上线前至少检查：

- 高风险工具是否全部经过工具网关；
- 风险分级是否由程序计算；
- 审批记录是否持久化；
- 工具执行参数是否和审批计划一致；
- 审批是否有有效期；
- 拒绝后是否会停止后续动作；
- 执行失败后是否会重新进入正确状态；
- 审计日志是否能还原计划、批准人、工具参数和结果。

如果任何一项只能靠“模型应该会这么做”保证，就还不能上线。

## 最终判断

```text
只读：可自动
低风险写：策略授权
高风险写：人工确认
不可逆动作：强制确认
需要恢复：状态机
```

人机确认是运行时控制，不是 Prompt 风格问题。
